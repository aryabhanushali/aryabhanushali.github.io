What Is Soup?

An exploration of top-down and bottom-up concept formation, incremental learning, classification, and case-based reasoning applied to the most disputed bowl of liquid in culinary history.

Part Two

Incremental Concept Learning

Incremental concept learning builds a concept definition one labeled example at a time. The algorithm asks two questions per example: (1) Is this an example of the concept? and (2) Does it fit the current definition? The intersection of those answers determines whether to generalize, specialize, or do nothing. Crucially, specific named heuristics guide how to generalize or specialize — they navigate the otherwise vast space of possible concept revisions.

Generalization Heuristics
Drop-link: Drop a required link absent in a new positive example.
Climb-tree: Replace a specific value with a more general parent using background knowledge.
Enlarge-set / Variabilize: Broaden a value's range or replace a constant with a variable.
Specialization Heuristics
Require-link: Make required a feature the concept has but the negative example lacks — the distinguishing link.
Forbid-link: Explicitly forbid a link that appears in a negative near-miss but must never appear in the concept.
Algorithm Example is soup? YES → Fits model? YES = Do nothing | NO = Generalize Example is soup? NO → Fits model? YES = Specialize | NO = Do nothing
POSITIVE #1 Chicken Noodle Soup Is soup? YES → No model yet → INITIALIZE
⊕ Initialize

The first positive example seeds the model. Every observed property becomes a required link. The model starts maximally specific — every feature of the first soup is assumed necessary.

Hot broth · savory · liquid-dominant · has solids · broth base

POSITIVE #2 Gazpacho Is soup? YES → Fits model? NO (cold ≠ hot) → GENERALIZE
⊕ Drop-link Heuristic

Gazpacho is cold — it doesn't satisfy "temp: hot," a link that is in the current model. Since gazpacho is a soup, we must generalize. The drop-link heuristic applies: the required link "must-have-temp" is present in the model but cannot be satisfied by this positive example, so we drop it. Temperature is no longer a required feature of soup.

Why not climb-tree? Climb-tree would require a background taxonomy where "hot" has a parent node (e.g., hot → any-temp). That could apply here, but drop-link is the simpler and more direct choice — temperature is simply not a defining feature of soup at all, so we remove the link entirely rather than generalizing its value.
NEGATIVE #1 Oatmeal Is soup? NO → Fits model? NO (fails liquid-dom) → DO NOTHING
○ Do Nothing

Oatmeal is NOT a soup, and the model already correctly rejects it — oatmeal is solid-dominant, failing the "must-have-texture: liquid-dom" link. No revision needed. This confirms that link is doing real work.

POSITIVE #3 Miso Soup Is soup? YES → Fits model? NO (solids: barely any) → GENERALIZE
⊕ Drop-link Heuristic

Miso soup is definitely a soup, but often contains almost no solid content. The model requires "solids: yes," but miso has no such link. Classic drop-link trigger: a link present in the current model is absent from a positive example — so we drop it. Solids become optional.

NEGATIVE #2 Hot Tea with Tea Leaves Is soup? NO → Fits model? YES (near-miss!) → SPECIALIZE
⊖ Require-link + Forbid-link

Hot tea is a near-miss — it is liquid-dominant and savory-adjacent, so the current model would accept it as soup. But it is NOT soup. We must specialize. Require-link: soup has a broth/stock base; tea does not — require it. Forbid-link: tea is a beverage; explicitly forbid that role.

NEGATIVE #3 Hot Chocolate with Marshmallows Is soup? NO → Fits model? NO (sweet + beverage) → DO NOTHING
○ Do Nothing

Hot chocolate fails two constraints: it is sweet (not savory) and it is a beverage (forbidden). The model correctly rejects it without any modification. This confirms both the savory link and the forbid-beverage link are doing real work in the concept definition.

Final Learned Model — Node-Link Representation of Soup

⚠ Base Constraint — Would Be Dropped

Tomato Bisque, Clam Chowder, Corn Chowder, and Vichyssoise are all unambiguously soups — yet none use a broth or stock as their primary base. They are cream- or dairy-based. Adding any one of them as the next positive example triggers drop-link on the must-have-base: broth/stock link, since that link is present in the model but absent in the new positive example. The base constraint disappears from the model entirely, leaving only: savory · liquid-dominant · ¬ beverage.

⚠ Liquid-Dom Constraint — Genuinely Ambiguous

Gumbo and Stew are classified as soup in Part 1, yet both are notably thick and solid-heavy. If added as positive examples, they would challenge the must-have-texture: liquid-dominant link — potentially forcing a second drop-link that removes the texture constraint entirely. This would leave an extremely weak final model (only: savory · ¬ beverage) that could no longer exclude oatmeal, porridge, or chili. This reveals the real tension: the model learned liquid-dominance from a negative example (oatmeal), but it would be forced to unlearn it if gumbo or stew are accepted as soups.

Part Three

Classification Approach

Where incremental concept learning produces a narrative model, classification takes a more structured approach: select a set of discriminating parameters, assign values to known examples, then build a decision tree that can classify novel cases. The five parameters below were chosen for maximum discriminating power.

P1
Primarily Liquid?
Liquid exceeds 50% of total volume by mass/perception. Excludes thick pastes, solids-in-sauce.
P2
Savory Flavor?
Predominantly savory (umami/salty), not sweet or neutral-as-a-dessert.
P3
Standalone Dish?
Served and consumed as a meal component or meal itself — not a beverage, sauce, or ingredient.
P4
Broth or Stock Base?
The liquid element is a broth, stock, water-based cook, or vegetable liquid — not milk, cream, or syrup alone.
P5
Spooned from Bowl?
Consumed with a spoon from a bowl or deep vessel — not drunk directly, eaten with fork/hands, or spread.

Part Four

Are Grits Soup?

Grits: Ground Corn, Cooked in Liquid, Served Warm

A Southern staple — dried corn hominy simmered in water, broth, or milk until thickened to a porridge. Savory or sweet depending on preparation. Served in a bowl and eaten with a spoon.

Incremental Concept Learning
Not Soup.

The final ICL model requires a dish to be primarily liquid, savory, and a standalone dish. Grits satisfies savory (when prepared traditionally) and standalone dish — but it fails the critical liquid-dominant constraint. Like oatmeal, which was the negative example that locked in that constraint, grits are a grain-thickened porridge where the solid content dominates. The liquid is absorbed into the corn and becomes the vehicle for the grain, not a surrounding medium. The ICL model, which specialized on oatmeal to require liquid dominance, would classify grits identically.

Classification Tree
Not Soup.

Walking down the classification tree:

P1 — Primarily Liquid? No.
Grits are thick. Once cooked, the solid corn content dominates; the dish would not pour freely. This fails the very first branch, immediately classifying grits as not soup without needing to evaluate P2 or P3. The tree reaches a leaf node at the first decision point — the same outcome as oatmeal and mashed potatoes.

Case-Based Reasoning
Not Soup.

The most similar case in our library is Oatmeal. Both are:

Unanimous Verdict

All three cognitive frameworks — incremental concept learning, classification, and case-based reasoning — agree: grits are not soup. The consistent failure point is liquid dominance. Grits are porridge: a grain-thickened hot dish that shares a bowl and a spoon with soup, but is structurally defined by absorption rather than immersion.
